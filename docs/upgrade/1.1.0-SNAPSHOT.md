## Configuración

### 1. Configuración de nuevos roles en Keycloak

En nuevas instalaciones, estos roles se crean automáticamente al importar el realm por defecto de `sgi-auth`.

En instalaciones existentes, deben crearse manualmente e incluirse en los grupos correspondientes.

#### 1.1 Roles y permisos en Keycloak

##### Tipos de confidencialidad (`CSP-TCONF-*`)

| Rol | Permiso |
|---|---|
| `CSP-TCONF-C` | Crear tipos de confidencialidad |
| `CSP-TCONF-E` | Editar tipos de confidencialidad |
| `CSP-TCONF-B` | Desactivar tipos de confidencialidad |
| `CSP-TCONF-R` | Reactivar tipos de confidencialidad |
| `CSP-TCONF-V` | Consultar tipos de confidencialidad (solo lectura) |

##### Tipos de descriptor de grupo (`CSP-TDESG-*`)

| Rol | Permiso |
|---|---|
| `CSP-TDESG-C` | Crear tipos de descriptor de grupo |
| `CSP-TDESG-E` | Editar tipos de descriptor de grupo |
| `CSP-TDESG-B` | Desactivar tipos de descriptor de grupo |
| `CSP-TDESG-R` | Reactivar tipos de descriptor de grupo |
| `CSP-TDESG-V` | Consultar tipos de descriptor de grupo (solo lectura) |

##### Tipos de grupo (`CSP-TGIN-*`)

| Rol | Permiso |
|---|---|
| `CSP-TGIN-C` | Crear tipos de grupo |
| `CSP-TGIN-E` | Editar tipos de grupo |
| `CSP-TGIN-B` | Desactivar tipos de grupo |
| `CSP-TGIN-R` | Reactivar tipos de grupo |
| `CSP-TGIN-V` | Consultar tipos de grupo (solo lectura) |

#### 1.2 Pantallas afectadas

| Grupo de roles | Pantallas |
|---|---|
| `CSP-TCONF-*` | CSP > Configuración > Tipos de confidencialidad |
| `CSP-TDESG-*` | CSP > Configuración > Tipos de descriptor de grupos |
| `CSP-TGIN-*`  | CSP > Configuración > Tipos de grupo |

#### 1.3 Aplicación manual en instalaciones existentes

1. Acceder a la **consola de administración de Keycloak**.
2. Seleccionar el realm **SGI**.
3. Navegar a **Roles**.
4. Crear cada uno de los roles listados:
   - Hacer clic en **Add role** e introducir el nombre exacto.
5. Navegar a **Groups**.
6. Para cada grupo al que se quiera dar acceso (p.ej. `SYSADM-CSP`, `ADMINISTRADOR-CSP`, ...):
   - Ir a la pestaña **Role Mappings**.
   - Seleccionar en **Available Roles** el rol o roles que correspondan a los permisos que se quieran otorgar.
   - Hacer clic en **Add selected**.

## APIs de integración

En esta versión se amplía la API del **Sistema de Gestión de la estructura Orgánica (SGO)** con los endpoints necesarios para recuperar las **unidades de vinculación** ([#55](https://github.com/herculesproject/sgi/issues/55)). La definición completa puede consultarse en detalle aquí:

- 📄 OpenAPI: [sgo-openapi-definition.yml](../apis-integracion/openapi/sgo-openapi-definition.yml)
- 🌐 Swagger SGO: [Ver API SGO](https://herculesproject.github.io/sgi/apis-integracion/swagger/sgo.html)

### Endpoints añadidos

| Método | Endpoint | Descripción |
|---|---|---|
| `GET` | `/unidades-vinculacion` | Lista de unidades de vinculación filtrada (RSQL), ordenada y paginada. |
| `GET` | `/unidades-vinculacion/{id}` | Detalle de una unidad de vinculación por su identificador. |
| `GET` | `/public/unidades-vinculacion` | Versión pública (sin autenticación) del listado. |
| `GET` | `/public/unidades-vinculacion/{id}` | Versión pública (sin autenticación) del detalle. |

> [!NOTE]
> Estos endpoints son **opcionales**. Sólo es necesario implementarlos si se habilita la gestión de unidades de vinculación en el módulo CSP.

## Procedimiento de actualización

1. Aplicar en los ficheros `values.yaml` de cada entorno los cambios de configuración descritos en la sección [Configuración](#configuración).
2. Ejecutar el despliegue siguiendo el [procedimiento estándar](common/deployment.md), utilizando el tag de la versión **`1.1.0-SNAPSHOT`**.

## Validación de la actualización

Tras completar el despliegue, validar que la actualización se ha aplicado correctamente:

1. Verificación de versiones:

    Comprobar que los deployments están utilizando la versión de imagen esperada:

    ```bash
    kubectl get deployments.apps -n <namespace> -o wide
    ```

2. Estado de los recursos:

    Verificar que todos los pods asociados al despliegue se encuentran en estado `Running` y `Ready`:

    ```bash
    kubectl get pods -n <namespace>
    ```

3. Validación funcional:

    Confirmar el correcto funcionamiento de los servicios afectados por la actualización, prestando especial atención a los cambios incluidos en el [CHANGELOG](../../CHANGELOG.md#no-publicado) de esta versión.
