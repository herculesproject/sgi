## Acciones previas al despliegue

> [!IMPORTANT]
> Realizar un **backup de la base de datos de Keycloak** antes de desplegar. La migración de
> esquema que Keycloak 26 ejecuta en el primer arranque **no es reversible** y el backup es el
> único camino de rollback.

1. Realizar el **backup** de la base de datos de Keycloak.
2. **(Solo Oracle)** Comprobar que la base de datos es **Oracle 19c o superior** (Keycloak 26 no soporta versiones anteriores)
3. **Preparar el `values.yaml`** al contrato de Keycloak 26, según [Configuración del `values.yaml`](#configuración-del-valuesyaml). Si la implantación usa los mappers SAML basados en JavaScript, añadir además sus variables de activación (ver [Mappers SAML](#mappers-saml)).
4. **Inventariar los IdP SAML** configurados y sus mappers, para validarlos tras el despliegue.

## Despliegue

1. Actualizar el repositorio de Helm del SGI a la última versión (se ha modificado para dar soporte a Keycloak 26). Ver [Actualizar repositorio de Helm](common/deployment.md#actualizar-repositorio-de-helm).
2. Ejecutar el despliegue siguiendo el [procedimiento estándar](common/deployment.md), utilizando el tag de la versión **`1.3.0-SNAPSHOT`**.

## Configuración posterior al despliegue

Estas acciones se realizan con Keycloak 26 ya desplegado. En **instalaciones nuevas**, el realm por defecto ya las incluye. En una **actualización**, el realm ya existe en la base de datos y el import se salta, por lo que deben aplicarse manualmente desde la consola de administración de Keycloak (realm **SGI**).

### Atributos de usuario (User Profile)

Desde Keycloak 23, el *User Profile* declarativo exige declarar los atributos que maneja el realm.

1. Ir a **Realm settings → User profile**.
2. En la pestaña **Attributes groups**, pulsar **Create attributes group** y crear el grupo:
   - **Name**: `sgi`
   - **Display name**: `SGI`
   - **Display description**: `Atributos específicos del SGI`
3. En la pestaña **Attributes**, pulsar **Create attribute** y crear los atributos:
   - **`userRefId`** (Display name: `User Ref Id`)
   - **`investigador`** (Display name: `Investigador`)

   Para ambos atributos configurar:
   - **Attribute group**: `sgi`
   - **Who can edit?**: solo **Admin** (el usuario no debe poder editarlos, los rellena el IdP).
   - **Who can view?**: indiferente (el SGI los obtiene del token).

Los atributos declarados aparecen en el **detalle del usuario** (agrupados bajo "SGI"), ya no se muestran en la pestaña *Attributes*.

Para admitir atributos adicionales existen dos opciones: 
 - declararlos como en el paso 3 (en el grupo **SGI** o en uno nuevo)
 - poner **Unmanaged Attributes** en **Enabled** para permitir cualquier atributo no declarado en el *User Profile*.
 
### Logout del cliente `front`

1. Ir a **Clients → `front` → Settings**.
2. En **Valid post logout redirect URIs**, indicar **`+`**.

> [!NOTE]
> Desde Keycloak 24, al cerrar sesión con redirección Keycloak valida que la URL de retorno esté registrada. El valor `+` autoriza como destinos de logout las mismas **Valid redirect URIs** del cliente.

### Mapper `Nonce backwards compatible` del cliente `front`

1. Ir a **Clients → `front` → Client scopes** y seleccionar **`front-dedicated`**.
2. En la pestaña **Mappers**, pulsar **Add mapper → By configuration** y elegir **Nonce backwards compatible**.
3. Asignarle un **nombre** (p.ej. `nonce-backwards-compatible`) y guardar.

> [!NOTE]
> Desde Keycloak 24, las versiones anteriores de `keycloak-js` ya no son compatibles con la gestión del `nonce` del ID Token. La webapp está fijada a una de esas versiones (21.1.2) porque su versión de Angular no admite un adaptador más moderno. Este mapper restablece el comportamiento que esos adaptadores esperan, de modo que el login funcione sin desactivar la validación del `nonce`.

## Validación de la actualización

1. Verificar el estado del despliegue (versiones de imagen y pods en `Running`/`Ready`) según
   [Verificación del despliegue](common/deployment.md#3-verificación-del-despliegue).
2. Validación funcional:
   - Login desde la webapp verificando que asigna los grupos esperados y logout (con redirección post-logout).
   - Los servicios Spring validan el JWT sin errores de JWKS.
   - Claims `user_ref_id` e `investigador` presentes en el token.


## Detalles técnicos de la versión
### Configuración del `values.yaml`

La siguiente tabla resume los cambios necesarios en el `values.yaml` para adaptar la configuración de Keycloak 13 a Keycloak 26.

| Antes (KC13, WildFly) | Ahora (KC26, Quarkus) | Notas |
|---|---|---|
| `settings.keycloakUser` / `keycloakPassword` | Sin cambios | Ahora se traducen a `KC_BOOTSTRAP_ADMIN_USERNAME` y `KC_BOOTSTRAP_ADMIN_PASSWORD` |
| `settings.proxyAddressForwarding: "true"` | `settings.proxyHeaders: "xforwarded"` | Cabeceras del proxy TLS a confiar |
| `settings.keycloakMigrationStrategy` | *(se elimina)* | El import se controla con `settings.importMode` |
| `DB_VENDOR` (`mssql`/`oracle`/`postgres`) | `KC_DB` | Mismo valor |
| `DB_ADDR` + `DB_PORT` + `DB_DATABASE` | `KC_DB_URL` | Se combinan en una única URL JDBC |
| `DB_SCHEMA` | `KC_DB_SCHEMA` | |
| `DB_USER` / `DB_PASSWORD` | `KC_DB_USERNAME` / `KC_DB_PASSWORD` | |

Las variables de BD (`KC_DB*`) van en `extraEnvVariables`. Construcción de `KC_DB_URL` según el vendor:

- **SQL Server** — `jdbc:sqlserver://<host>:<port>;databaseName=<db>`
- **Oracle (Service)** — `jdbc:oracle:thin:@//<host>:<port>/<service>`
- **Oracle (SID)** — `jdbc:oracle:thin:@<host>:<port>:<sid>`
- **PostgreSQL** — `jdbc:postgresql://<host>:<port>/<db>`

Para pasar propiedades adicionales al driver (timeouts, cifrado, etc.) se puede usar `KC_DB_URL_PROPERTIES` (p.ej. en Oracle: `?oracle.net.CONNECT_TIMEOUT=61000`).

Ejemplo orientativo:

```yaml
sgi-keycloak:
  settings:
    keycloakUser:
      valueFrom:
        secretKeyRef:
          name: sgi-keycloak-secret
          key: keycloak_admin_user
    keycloakPassword:
      valueFrom:
        secretKeyRef:
          name: sgi-keycloak-secret
          key: keycloak_admin_password
    proxyHeaders: "xforwarded"
    httpEnabled: true
    importMode: "if-absent"
  extraEnvVariables:
    - name: KC_DB
      value: "mssql"
    - name: KC_DB_URL
      value: "jdbc:sqlserver://<host>:1433;databaseName=<db>"
    - name: KC_DB_SCHEMA
      value: "AUTH_SGI"
    - name: KC_DB_USERNAME
      valueFrom:
        secretKeyRef:
          name: sgi-auth-db-credentials
          key: db_username
    - name: KC_DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: sgi-auth-db-credentials
          key: db_password
```

> [!NOTE]
> `settings.importMode` admite dos valores:
> - `if-absent` (por defecto): importa el realm únicamente si no existe.
> - `none`: desactiva la importación del realm.

### Mappers SAML

Se han añadido dos mappers declarativos (`JSON Rules Group Importer`, `Prefix Match & Strip Attribute Importer`) que sustituyen a los mappers basados en JavaScript (`Script to Group`, `Script to Attribute`) que pasan a estar **desactivados por defecto**.

Si la implantación usa los mappers legacy para que sigan funcionando tras la actualización hay que reactivarlos añadiendo en `extraEnvVariables` del `values.yaml` antes del despliegue:

```yaml
- name: KC_SPI_IDENTITY_PROVIDER_MAPPER__SAML_SCRIPT_GROUP_IDP_MAPPER__ENABLED
  value: "true"
- name: KC_SPI_IDENTITY_PROVIDER_MAPPER__SAML_SCRIPT_ATTRIBUTE_IDP_MAPPER__ENABLED
  value: "true"
```

No obstante, se recomienda migrar a los nuevos mappers declarativos, ya que evitan la ejecución de JavaScript y ofrecen una solución más segura y mantenible de cara a futuras actualizaciones.

Para minimizar el alcance de los cambios durante esta actualización, se recomienda realizar primero la actualización de versión y validar que todo funciona correctamente. Una vez validada la actualización, puede abordarse la migración de los mappers siguiendo el proceso descrito en [`docs/keycloak-saml-mappers.md`](../keycloak-saml-mappers.md).

Una vez completada la migración y verificado que los nuevos mappers funcionan correctamente, se recomienda desactivar de nuevo los mappers basados en JavaScript eliminando las variables anteriores de `extraEnvVariables` (o estableciendo su valor a `"false"`).

No hay cambios en el modelo de datos de los servicios del SGI en esta versión.

Keycloak migra su propio esquema en el primer arranque; esta migración es la que motiva realizar el backup de las [acciones previas](#acciones-previas-al-despliegue).
